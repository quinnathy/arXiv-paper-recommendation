§ 0 System Overview

The recommendation engine surfaces three papers per user per day from the ArXiv corpus (~2M papers). It is divided into two phases: an offline batch pipeline that runs nightly, and an online serving pipeline that executes once per user at delivery time. All paper and user representations live in a shared 768-dimensional embedding space produced by SPECTER2, a scientific-text encoder pretrained on citation graphs.

The core algorithmic idea is that a user's taste can be approximated as a single unit vector in this same space, initialized from their topic selections and refined by accumulated feedback. Recommendation becomes geometric: find papers whose embeddings are closest to the user's vector.

§ 1 Offline Batch Pipeline Nightly · One-time

1.1 — Corpus Preparation. The ArXiv dataset is loaded from its Kaggle JSON snapshot. Each record provides an arxiv_id, title, abstract, categories (space-separated ArXiv tags such as cs.LG), and update_date. The text input to the encoder is the concatenation title. abstract, which gives SPECTER2 the citation-graph context it was trained on.

1.2 — Embedding. SPECTER2 encodes each paper into a 768-dimensional vector. All vectors are L2-normalized to unit length immediately after encoding, so that cosine similarity reduces to a dot product — a property exploited throughout the pipeline for efficiency.

ei = normalize( SPECTER2( titlei + ". " + abstracti ) )
→ result: matrix E ∈ ℝN×768, each row unit-norm

The full embedding matrix occupies N × d × 4 bytes ≈ 5.9 GB in float32. This is held in RAM for the lifetime of the serving process. A float16 variant halves this to ~1.5 GB with negligible loss in dot-product accuracy.

1.3 — k-Means Clustering. MiniBatchKMeans partitions the N paper embeddings into k = 500 clusters. The purpose of clustering is not topic discovery for its own sake — it is a spatial index that restricts KNN search to a small fraction of the corpus at query time. With k = 500 and N = 2M, each cluster contains on average 4,000 papers. KNN within a single cluster requires 4,000 × 768 ≈ 3M multiply-adds, compared to 1.5B for the full corpus — a ~500× reduction.

cluster_ids, centroids = MiniBatchKMeans(E, k=500)
→ centroids C ∈ ℝ500×768, cluster_ids ∈ ℤN

1.4 — Category Centroids. Separately from k-means, one centroid per ArXiv category label is computed as the mean (then normalized) of all paper embeddings belonging to that category. These are stored in a lookup table and used exclusively for user cold-start initialization. This decoupling is important: category centroids are stable and interpretable by label; k-means cluster centroids are anonymous and may not align neatly with human-readable topics.

ccat = normalize( mean({ ei : categoryi = cat }) )
→ stored as dict { "cs.LG": vector, "cs.CL": vector, … }
§ 2 User State: Initialization and Updates Live · Per-event

2.1 — Cold-Start Initialization. When a new user selects a set of topic interests T = {t₁, t₂, …}, their initial embedding is the normalized sum of the corresponding category centroids. This places the user at the geometric centroid of their interests in embedding space — a point equidistant from all selected topic neighborhoods. If a user selects two closely related topics (e.g. "NLP" and "computational linguistics"), the average is naturally pulled toward that region, correctly reflecting stronger interest density there.

u0 = normalize( Σt ∈ T ct )
→ single 768-d unit vector; user lives between all selected topic neighborhoods

2.2 — Feedback Weights. Three explicit feedback signals are supported. Implicit non-interaction (paper was served but ignored) is not treated as a skip — the user may simply not have opened the app. Only deliberate actions trigger updates.

SignalWeight wRationale
like+1.0Standard positive signal; paper was interesting
save+1.5Stronger intent; user intends to return to this paper
skip−0.3Weak negative; asymmetric to prevent over-correction

2.3 — EMA Update Rule. Each feedback event moves the user embedding toward or away from the paper's embedding using an Exponential Moving Average with learning rate α = 0.15. The result is renormalized to maintain unit length. The small α ensures no single paper can dramatically redirect the user's taste vector; it takes several consistent signals to meaningfully shift the embedding.

u' = normalize( (1 − α) · u + α · w · epaper )

where: u = current user embedding (unit vector)
e_paper = embedding of paper acted on (unit vector)
w = feedback weight ∈ {+1.5, +1.0, −0.3}
α = 0.15 (EMA learning rate)
Edge case If a skip perfectly cancels the current user vector (norm of raw update < 1e-8), the update is discarded and the embedding is unchanged. In practice this is geometrically impossible with α = 0.15, but the guard is retained for safety.
§ 3 Online Serving Pipeline Daily · Per-user

3.1 — Cluster Selection. Given the user embedding u, the two nearest cluster centroids are identified by computing dot products with all k = 500 centroids (a 500 × 768 matrix-vector product, ~0.4M FLOPs, negligible). The top-2 clusters are searched to improve recall at cluster boundaries — papers near the edge of the user's primary cluster may rank higher than papers at the core of a nearby secondary cluster.

sims = C · u    ← dot product, shape (500,)
target_clusters = argsort(sims)[::−1][:2]

3.2 — Exact KNN within Clusters. All paper embeddings belonging to the selected clusters are extracted from the global matrix E (boolean masking). Dot products are computed between this sub-matrix and the user vector u, producing a similarity score for every candidate paper. Already-seen papers are removed, and the top M = 40 candidates by similarity score are retained.

mask = cluster_ids ∈ target_clusters    ← ~8,000 papers
sims = E[mask] · u                 ← shape (~8000,)
candidates = top-M by sims, excluding seen_ids

3.3 — Re-ranking. Raw similarity scores are adjusted with a recency bonus. Papers published within the last 30 days receive the full bonus; it decays exponentially with age. ArXiv is a preprint server — a paper from last week is almost always more actionable than one from five years ago, even at identical similarity.

recency(p) = exp( −age_days(p) / 30 )
score(p) = sim(p, u) + 0.25 · recency(p)

3.4 — Diversity Filter. Candidates are iterated in descending score order. A paper is selected only if its k-means cluster has not already contributed a selection. This ensures the three served papers span different topic sub-regions — preventing the degenerate case where all three are near-duplicates from a single very dense cluster (e.g. three variations on the same hot sub-topic).

  1. Sort all M candidates by score descending.
  2. Iterate through sorted candidates; maintain a set of used cluster IDs.
  3. Add candidate to output only if its cluster ID is not in the used set.
  4. Stop when n = 3 papers have been selected.
  5. Mark all three as seen for this user; log scores and cluster IDs.
Complexity The full serving pipeline for one user: 500 dot products (cluster selection) + ~8,000 dot products (KNN) + O(M log M) sort (re-rank) + O(M) pass (diversity). Total: <1 ms on CPU. This is fast enough to run synchronously on a daily cron for any realistic number of users.
